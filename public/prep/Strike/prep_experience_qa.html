Bureau Veritas
BV ●○○ ▾
"You tested 100+ vulnerabilities across 25+ services. Walk me through how you managed that scale."
"It wasn't 100 separate engagements — it was sustained testing across a large AWS account with many service teams. The approach was to prioritize by attack surface: authentication, authorization, and data access paths first, because those chain into the most impactful findings. I tracked findings in structured reports, grouped by severity and service, and worked directly with each engineering team to validate exploitability before raising the priority. The 100+ number came from being systematic about what I documented — including lower-severity issues that were still worth tracking for pattern analysis."
Key point: Show that scale came from methodology, not from inflating counts. The pattern analysis angle (BV-013) is the most interesting part — bring it in if they probe.
BV ●●○ ▾
"You identified recurring auth anti-patterns across 25+ services. What was the pattern and what did you actually change?"
"The recurring issue was inconsistent authorization enforcement at the service boundary — services trusting caller identity based on internal headers that weren't actually validated, or applying access control checks at the UI layer but not the API layer. Once I saw it repeat across three or four services I stopped treating them as individual findings and went back to the engineering leads with a pattern writeup — here's the class of problem, here's why it keeps appearing, here's the design change that prevents it. The outcome was a secure-by-default template for how new services should handle inter-service auth."
Key point: This is BV-013. The value is systemic impact — not just finding bugs but changing how bugs get introduced. This maps directly to the JD's "scalable security capabilities" requirement.
BV ●●○ ▾
"You mention chaining low-severity findings into high-impact paths. Give me a concrete example."
"One example: individually, an overly verbose error message (informational), a misconfigured IAM permission on a service role (low), and a missing authorization check on an internal API path (low-medium) were each unremarkable. But together they formed a path — use the error message to enumerate a valid internal endpoint, use the misconfigured service role to get temporary credentials, and use the missing authz check to hit a privileged internal API that shouldn't be reachable. No single finding raised a flag. The chain was a privilege escalation. That's why I correlated across CloudWatch logs, code review, and API testing rather than treating each stream separately."
Key point: This is your BV-008 bullet in narrative form. Shows attacker mindset — the whole is greater than the sum of parts.
BV ●○○ ▾
"What was your threat modeling process at BV?"
"I reviewed existing threat model documents rather than building them from scratch — the service teams had done initial design documentation, and my job was to stress-test the assumptions. I focused on trust boundary violations, specifically cases where a component trusted another component's input without independent validation, and abuse cases that weren't in the original threat model — how an attacker with partial access could pivot, or how a legitimate user could trigger unintended behavior at scale. Findings went back to the engineering team with specific design recommendations, not just a list of problems."
Key point: BV-009. Shows you add to threat models rather than just ratifying them. The "abuse cases not in the original model" angle is the interesting differentiator.
BV ●●○ ▾
"You did log reviews to validate incident readiness. What were you specifically checking for?"
"Two things mainly: coverage and fidelity. Coverage — were the right events being logged at all? Authentication failures, privilege use, API errors, config changes. There are whole classes of attacker behavior that leave no trace if logging isn't instrumented for them. Fidelity — were the logs usable? Structured enough to query, retained long enough to be useful for an incident investigation, not so noisy that real signals would drown. In a few cases I found that a service was logging authentication events but not logging which resource was accessed — useful for detecting an attempt but not for understanding blast radius after the fact."
Key point: This is BV-019 in narrative. Directly maps to the SIEM uplift requirement in the JD. The coverage vs. fidelity framing is strong.
BV ●●● ▾
"You reviewed 120+ repos. How did you triage which ones to go deep on?"
"Not all 120 got the same depth — that would be impossible and the wrong approach anyway. I triaged by trust level and data exposure. Services that handled authentication, session management, or user data got full manual review. Services that were internal-only with no external-facing endpoints got lighter review focused on dependency scanning and obvious logic paths. I also used early findings as signals — if one service had a broken object-level authorization issue, I'd prioritize reviewing any service it interacted with, because the pattern tends to repeat across teams that share architectural decisions."
Key point: Shows judgment, not just volume. Triage methodology is what separates someone who has done scale work from someone who just claims it.
WinWin Labs
WWL ●●○ ▾
"Tell me about the critical authentication flaw you found and patched at WinWin Labs."
"It was a logic flaw in the authentication flow — not a credential weakness but a sequencing issue in the validation steps that allowed an attacker to bypass the full authentication check under specific conditions. The conditions weren't obscure, which is what made it critical. I found it during API testing and traced it back through the code to understand the root cause. The patch wasn't just fixing the one path — I worked with the developer to understand why the logic was structured that way and whether the same pattern existed elsewhere. We found one other endpoint with a similar issue."
Key point: The "found one other endpoint" part shows you think in patterns, not just single findings. That's the signal the CISO cares about for a product-embedded role.
WWL ●○○ ▾
"You used STRIDE at WinWin Labs. How did you apply it in practice?"
"STRIDE gave me a structured way to make sure I wasn't missing threat categories. For a web platform the useful ones were Spoofing (can someone impersonate another user or service?), Tampering (can they modify data in transit or at rest?), and Elevation of Privilege (can a lower-trust user reach higher-trust functionality?). I went through each component in the architecture and systematically asked those questions rather than relying on intuition. The output was a prioritised list of threat scenarios that mapped to specific remediation tasks — not just a theoretical document."
Key point: Show STRIDE as a practical tool, not a checkbox. The "prioritised list that maps to remediation tasks" framing shows you close the loop.

ZOHO ●●○ ▾
"You developed POCs for 50+ vulnerabilities in C++ codebases. What were the most common classes?"
"Buffer overflows and use-after-free were the most common in the native code. The C++ codebase had grown over years and had inconsistent memory management patterns — some components used smart pointers consistently, others relied on manual allocation. The vulnerabilities tended to cluster around the boundaries between those patterns, where an assumption about object lifetime in one module didn't match the actual lifetime in another. The POC work was important because it separated theoretical vulnerabilities from exploitable ones — a lot of buffer overflows in practice can't be reached in a way that gives you useful control."
Key point: The "exploitable vs. theoretical" distinction is mature thinking. Shows you don't just find issues — you understand their actual severity.
ZOHO ●○○ ▾
"You implemented detection logic for OWASP Top 10 and CWE/SANS Top 25. Walk me through how that worked in an endpoint security product."
"The product was scanning enterprise endpoints for vulnerabilities and misconfigurations. My job was to translate the OWASP Top 10 and CWE/SANS Top 25 categories into detectable conditions on the endpoint — what configuration state, file permission, registry key, or process behavior would indicate a vulnerability in that class. For injection classes that meant looking at server configurations that would allow unsanitized input paths. For access control classes it meant checking permission models on web server directories and application config files. The output was a detection rule that the product could run and report on, with a CVE or CWE reference for the finding."
Key point: Shows you understand both the offensive classification (OWASP/CWE) and the defensive implementation (detection rules). Relevant for the SIEM alerting work at Strike.
Projects & Disclosures
PROJECT ●●● ▾
"Tell me about Code Mapper. What problem were you actually solving?"
"The core problem is that LLMs have a context window — they can't hold an entire codebase in memory at once, so naive AI security tools miss vulnerabilities that span multiple files. A taint flow from user input in one file to a dangerous sink in another file is exactly what gets missed. Code Mapper breaks the problem apart across five specialized agents: one models the domain and trust boundaries, one does semantic reasoning about code behavior, one classifies data sensitivity, one maps the attack terrain, and one traces taint propagation. Each agent handles a scoped problem that fits in context, then the results get correlated. It found attack surfaces on a test target that a commercial SAST tool didn't have visibility into."
Key point: Lead with the problem (context window limitation), not the solution. This shows systems thinking. For a CISO the interesting part is "found things commercial tools missed."
DISCLOSURE ●○○ ▾
"You disclosed two vulnerabilities in ManageEngine. How did you find them and what was the process?"
"The path hijacking one came from testing the agent installer — looking at what executables it called and whether the paths were fully qualified. If a Windows service calls an unquoted executable path and an attacker can write to an intermediate directory, they can substitute their own binary. ManageEngine's VMP agent had this pattern in a path with write access available to lower-privileged users. I documented the reproduction steps, the privilege level required, and the impact, and submitted through their security contact. They acknowledged it and published an advisory. The logging issue I found while reviewing how the agent reported errors — it was including command output in log entries that were accessible to non-admin users."
Key point: Walk through the discovery method and the disclosure process. Shows you can handle responsible disclosure professionally, which matters for a fintech company with regulatory exposure.
PROJECT ●●○ ▾
"The SINES framework — walk me through what it does and why you built it."
"The idea came from work I did building a honeypot detection system — embedding webhook triggers in documents to detect when they were accessed inside internal infrastructure, funneling events through Tines into Splunk for alerting. I wanted to extend that into automated response: not just alert that something suspicious happened, but kick off artifact collection automatically so you have forensic data before it's overwritten. SINES is a client-server DFIR framework — the server receives the alert trigger, the client agents on Windows and Linux endpoints collect 43 predefined forensic artifacts and send them back. The goal was to compress the time between detection and having usable evidence."
Key point: This is your SIEM/IR story. The Chegg honeypot context gives the motivation. The "compress time between detection and evidence" framing is the right way to describe the value.
