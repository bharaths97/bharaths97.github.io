01 · Language

Solidity

The primary language for writing smart contracts on Ethereum and any EVM-compatible chain. Statically typed, compiled down to EVM bytecode. Syntax is intentionally similar to JavaScript — accessible but with significant footguns if you don't understand how the EVM handles state and money.

Contract Structure

A contract in Solidity is structurally similar to a class in object-oriented languages. It has:

  • State variables — stored permanently on-chain. Reading them costs gas. Writing them costs more.
  • Functions — the interface. Can read state, write state, or do both. Can receive or send Ether.
  • Events — logs emitted to the blockchain. Cheaper than storage. Used by frontends and indexers to track activity.
  • Modifiers — reusable conditions applied to functions. onlyOwner is the most common pattern — gates a function to a specific address.
  • Constructor — runs exactly once at deployment. Sets initial state.

Visibility Modifiers

Every function and variable has a visibility modifier. These control who can call or read it.

public

Callable from anywhere — inside the contract, other contracts, and external wallets. Automatically generates a getter for state variables.

external

Only callable from outside the contract. More gas-efficient than public for functions that only need to be called externally. Cannot be called internally by the contract itself.

internal

Only callable from inside the contract or contracts that inherit from it. The default for state variables if no modifier is specified.

private

Only callable from inside the same contract. Not inherited. Note: "private" on-chain still means the value is readable by anyone who inspects the blockchain — it just can't be called programmatically.

Function State Mutability

  • view — reads state, doesn't modify it. No gas cost when called externally.
  • pure — neither reads nor modifies state. Purely computational. No gas cost when called externally.
  • payable — can receive Ether. Functions without this modifier will reject any Ether sent to them.
  • No keyword — can read and write state. Costs gas.

The Immutability Point

Once a contract is deployed its code cannot be changed. There is no patch, no hotfix, no rollback. If there is a bug, the options are: (1) deploy a new contract and migrate, (2) use an upgradeable proxy pattern — but that must be designed in from the start. This is the fundamental difference between smart contract security and traditional AppSec. Pre-deployment review is not a nice-to-have.
02 · Runtime

The Ethereum Virtual Machine

The EVM is the sandboxed runtime that executes smart contract bytecode. Every node in the Ethereum network runs the same EVM and must arrive at the same result for every transaction — that determinism is how consensus works.

Key Concepts

  • Bytecode: Solidity compiles to low-level EVM bytecode — stack-based opcodes (PUSH, ADD, SLOAD, SSTORE etc.). The EVM executes this, not the Solidity source.
  • Stack-based execution: The EVM uses a last-in-first-out stack for computation. 1024 item depth limit.
  • Gas: Every opcode has a gas cost. Gas is the metered execution unit — it prevents infinite loops and DoS by requiring every operation to be paid for by the caller. If execution runs out of gas, the entire transaction reverts with no state changes.
  • Accounts: Two types — Externally Owned Accounts (wallets, controlled by private keys) and Contract Accounts (smart contracts, controlled by code).
  • Storage vs Memory: Storage is permanent, expensive on-chain state. Memory is temporary, cheaper, exists only for the duration of a function call.
  • msg.sender: The address that called the current function. Fundamental to access control — most permission checks read msg.sender.
  • msg.value: The amount of Ether (in wei) sent with the transaction.

EVM-Compatible Chains

The EVM specification is open. Many chains implement it, meaning Solidity contracts deploy identically across all of them:

  • Polygon, Arbitrum, Optimism, Base — Ethereum L2s
  • BNB Smart Chain, Avalanche C-Chain, Fantom
  • This matters for Strike: if they ever build smart contract features, they're most likely deploying to an EVM chain.
03 · Token Standard

ERC-20 — Fungible Tokens

ERC stands for Ethereum Request for Comments — it's the process by which standards are proposed and adopted. ERC-20 is the standard interface for fungible tokens: every unit is identical and interchangeable. Think of it as the specification that every dollar-like token on Ethereum follows.

Fungible means: One token is worth exactly the same as any other token of the same type. 1 USDC = 1 USDC. Contrast with NFTs where each token is unique.

Required Functions (the ERC-20 Interface)

Function What it does
totalSupply()
→ uint256
view		 Returns the total number of tokens in existence. Read-only. No state change.
balanceOf(address)
→ uint256
view		 Returns how many tokens a given address holds. The core ledger lookup.
transfer(address, uint256)
→ bool
 Moves tokens from the caller's address to the destination. Emits a Transfer event. Returns true on success.
approve(address, uint256)
→ bool
 Authorises a spender address to withdraw up to a specified amount from the caller's balance. Does not move tokens immediately — just sets an allowance.
allowance(address, address)
→ uint256
view		 Returns the remaining amount a spender is allowed to spend on behalf of an owner. The current value set by approve.
transferFrom(address, address, uint256)
→ bool
 Moves tokens from one address to another on behalf of the owner — using the allowance set by approve. This is how decentralised exchanges and protocols interact with user balances.

Required Events

Event When it fires
Transfer(from, to, value) Every time tokens move between addresses, including minting (from = 0x0) and burning (to = 0x0).
Approval(owner, spender, value) Every time an allowance is set or changed via approve.

The approve / transferFrom Pattern — Why It Matters

This two-step pattern is how DeFi protocols interact with your tokens without you handing over your private key. You call approve to say "this contract can spend up to X tokens on my behalf," then the protocol calls transferFrom to actually move them. Understanding this pattern is foundational to understanding how token interactions work.

Optional but Common

  • name() — returns the token's human-readable name (e.g. "USD Coin")
  • symbol() — returns the ticker (e.g. "USDC")
  • decimals() — returns the number of decimal places (almost always 18 for ETH-like tokens, 6 for USDC)
04 · Token Standard

ERC-721 — Non-Fungible Tokens

ERC-721 is the standard for non-fungible tokens — each token has a unique ID and is not interchangeable with any other. Instead of tracking balances by address, ERC-721 tracks ownership by token ID.

Non-fungible means: Token #42 and token #43 from the same contract are distinct. Owning one does not mean owning the other. Think of it as a deed of ownership for a specific item.

Required Functions

Function What it does
balanceOf(address)
→ uint256
view		 Returns how many NFTs an address owns in this collection. Not which specific tokens — just the count.
ownerOf(uint256 tokenId)
→ address
view		 Returns the current owner of a specific token ID. The core ownership lookup.
transferFrom(address, address, uint256) Transfers ownership of a specific token ID from one address to another. Caller must be the owner or approved.
safeTransferFrom(address, address, uint256) Same as transferFrom but checks if the receiving address is a smart contract and if so, verifies it can handle ERC-721 tokens. Prevents tokens getting locked in contracts that don't know how to interact with NFTs.
approve(address, uint256) Approves a single address to transfer a specific token ID on the owner's behalf. Per-token approval, unlike ERC-20's per-amount approval.
getApproved(uint256 tokenId)
→ address
view		 Returns the approved address for a specific token ID.
setApprovalForAll(address, bool) Approves or revokes an operator to manage ALL tokens the caller owns in this collection. Broader than approve — used by marketplaces like OpenSea.
isApprovedForAll(address, address)
→ bool
view		 Checks if an operator is approved to manage all of an owner's tokens.

Optional but Widely Used

Function What it does
tokenURI(uint256 tokenId)
→ string
view		 Returns a URI pointing to the token's metadata (name, description, image). Usually points to IPFS or an HTTPS endpoint. The metadata itself lives off-chain — only the pointer is on-chain.
name()
→ string
view		 Collection name (e.g. "CryptoPunks").
symbol()
→ string
view		 Collection ticker symbol.

ERC-721 vs ERC-20 — Key Differences

ERC-20 ERC-721
Unit Amount (uint256 balance) Specific token ID (uint256 tokenId)
Ownership model address → balance tokenId → owner address
Interchangeable? Yes — 1 token = 1 token No — each token ID is unique
Approval scope Amount per spender Per token ID, or all-or-nothing via operator
05 · Token Standard

ERC-1155 — Multi-Token Standard

ERC-1155 is a more flexible standard that can handle both fungible and non-fungible tokens in a single contract. A single ERC-1155 contract can define multiple token types, each with its own ID — some can be fungible (many copies of the same item), some non-fungible (unique items).

Analogy: A single game item contract. Token ID 1 = health potion (fungible — you can have 50 of them). Token ID 2 = legendary sword #7 (non-fungible — unique item). One contract manages both.

Key Functions

Function What it does
balanceOf(address, uint256 id)
→ uint256
view		 Returns how many of a specific token ID an address holds. Takes both the address and the token ID — unlike ERC-20 which only needs the address.
balanceOfBatch(address[], uint256[])
→ uint256[]
view		 Batch version of balanceOf — check multiple owners and token IDs in a single call. Gas-efficient for querying many balances at once.
safeTransferFrom(address, address, uint256 id, uint256 amount, bytes) Transfer a specific amount of a specific token ID from one address to another.
safeBatchTransferFrom(address, address, uint256[], uint256[], bytes) Transfer multiple token types in a single transaction. Much more gas-efficient than multiple individual transfers.
setApprovalForAll(address, bool) Same as ERC-721 — approve an operator to manage all tokens the caller holds across all IDs in this contract.
isApprovedForAll(address, address)
→ bool
view		 Check if an operator is approved for all tokens.
uri(uint256 id)
→ string
view		 Returns metadata URI for a token ID. Can use {id} placeholder to return different URIs per token.
06 · Interface Detection

ERC-165 — Standard Interface Detection

ERC-165 is a meta-standard that lets contracts advertise which interfaces they implement. It answers the question: "Does this contract support ERC-721?" before you try to interact with it.

Without ERC-165, a contract calling another contract would have to blindly assume it implements the right functions — which can cause silent failures or locked tokens.

The Single Function

Function What it does
supportsInterface(bytes4 interfaceId)
→ bool
view		 Takes a 4-byte interface identifier (a hash of the function signatures in the interface) and returns true if the contract implements that interface. Every ERC-721 and ERC-1155 contract must implement this.
Why it matters: The safeTransferFrom functions in ERC-721 and ERC-1155 use ERC-165 to check if the receiving contract can handle NFTs before transferring. If the check fails, the transfer reverts — preventing tokens from being permanently locked in a contract that has no way to send them out.
07 · Synthesis

The Mental Model That Connects All of This

If the CISO goes deeper into blockchain topics, this is the framing that shows you understand the space conceptually, not just the terminology.

How to articulate it "The thing that struck me most about this space is that the standard AppSec safety net doesn't exist. In a web app, if you find a bug after deployment you push a hotfix. You rotate credentials. You roll back. In a smart contract, none of those options exist unless you designed an upgrade mechanism in from the start — and even then, that mechanism itself is an attack surface. The implication is that the cost of a pre-deployment review is trivially small compared to the cost of finding something after. That's a different security calculus than I've been working with."

The Standards as a Parallel to API Contracts

If it helps to connect this to your AppSec background: ERC standards are essentially ABI specifications — they define the function signatures, inputs, outputs, and events that any compliant implementation must expose. Interacting with a contract that claims to be ERC-20 but has a non-standard transfer implementation is the same class of problem as an API that documents one schema and returns another. The contract fulfils the interface but breaks the assumption.

Where You Would Go Next in the Course

Being transparent about this shows intellectual honesty, not weakness:

  • Reentrancy — contracts calling back into themselves before state updates complete
  • Integer overflow/underflow — arithmetic wrapping in older Solidity versions
  • Access control failures — unprotected mint or admin functions
  • Front-running — transaction ordering manipulation by miners or bots

You know these exist as a vulnerability class. You haven't studied the exploitation techniques yet. That's the honest line.
